## AWS IAM (Identity and Access Management)

- It is a service that helps you securely control who can access AWS resources and what actions they can perform.
- IAM is a **global service**

With IAM, you can:

* Create users
* Create groups
* Assign permissions
* Control access to AWS services
  
---

## Why Use IAM?

IAM helps you:

* Secure your AWS account
* Give users only the permissions they need
* Avoid sharing the root account
* Manage access easily

---

## IAM Components

### 1. IAM User

- An IAM User represents a person or application that needs access to AWS.

Example:

* Alice (Developer)
* Bob (Administrator)

Each user can have:

* Username
* Password (AWS Console)
* Access Keys (CLI/API)

---

### 2. IAM Group

- A Group is a collection of IAM users with the same permissions.
- Adding a user to a group automatically gives them the group's permissions.

Example:

| Group      | Permissions         |
| ---------- | ------------------- |
| Developers | EC2, S3             |
| Admins     | Full Access         |
| ReadOnly   | View resources only |


---

### 3. IAM Policy

- A Policy is a JSON document that defines permissions.

Example:

* Allow access to S3
* Deny deleting EC2 instances

Policies can be attached to:

* Users
* Groups
* Roles

---

### 4. IAM Role

- An IAM Role provides temporary permissions to AWS services or users.
- Roles do not have passwords or access keys.

Common examples:

* EC2 accessing S3
* Lambda accessing DynamoDB
* Cross-account access


---

## Authentication vs Authorization

| Authentication | Authorization    |
| -------------- | ---------------- |
| Who are you?   | What can you do? |
| Login          | Permissions      |

Example:

* Login using IAM User → Authentication
* Access S3 bucket → Authorization

---

## Root User vs IAM User

| Root User                           | IAM User                      |
| ----------------------------------- | ----------------------------- |
| Created when AWS account is created | Created by Root or Admin      |
| Full access                         | Limited access                |
| Cannot be restricted                | Permissions can be controlled |
| Use only for account-level tasks    | Use for daily work            |

> Never use the Root User for everyday activities.

---

## IAM Best Practices

* Enable Multi-Factor Authentication (MFA).
* Never share the Root User credentials.
* Follow the Principle of Least Privilege.
* Use Groups instead of assigning permissions to individual users.
* Rotate access keys regularly.
* Remove unused users and permissions.

---

## Example

Suppose a company has three employees:

* Alice – Developer
* Bob – Developer
* Charlie – Administrator

Create:

* **Developers** group → EC2 and S3 access
* **Admins** group → Full access

Add Alice and Bob to the Developers group, and Charlie to the Admins group.

This makes permission management simple and consistent.

---

## Key Points

* IAM controls access to AWS resources.
* IAM is a global service.
* Users represent people or applications.
* Groups help manage permissions for multiple users.
* Policies define what actions are allowed or denied.
* Roles provide temporary permissions.
* Avoid using the Root User for daily tasks.
